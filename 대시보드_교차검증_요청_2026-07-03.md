## 확인할 핵심 변경점
1. Apps Script 응답이 직접 payload이거나 `dashboardPayload`, `dashboard_payload`, `payload`, `data`, `dashboard` wrapper에 들어와도 프론트가 정상 렌더링하는지 확인.
2. 운영 원칙상 `WEATHER_OPS_ALLOW_SAMPLE=false`일 때 샘플 fallback이 켜지지 않고, API 오류가 명확히 표시되는지 확인.
3. `visuals.processedBulletByStore`, `visuals.systemTrend`, `visuals.openActionTrend`, `recovery.gapByStore`, `recovery.bulletByStore`를 누락 없이 수용하는지 확인.
4. 회복 큐에서 `processedRecoveryRate`와 `revenueRecoveryRate`를 함께 표시해 처리대수 회복과 매출 회복을 혼동하지 않는지 확인.
5. 모바일/데스크톱에서 신규 bullet/trend UI가 겹치거나 버튼/카드 텍스트가 깨지지 않는지 확인.
6. 인증, CSP, Apps Script token 전달, `fresh=1` 캐시 우회가 기존 보안/운영 원칙을 훼손하지 않는지 확인.
7. `script-src`와 `connect-src`가 운영에 필요한 최소 범위로 유지되는지 확인.
8. 최신 시트/Apps Script payload 변경이 대시보드에서 snake_case, camelCase, 일부 한글/레거시 필드 alias로 안전하게 흡수되는지 확인.

## 검토 제한
- 읽기 전용 검토만 수행하고 파일 수정, 커밋, 배포는 하지 마세요.
- 시트 또는 Apps Script 수정 제안은 가능하지만, 이번 검토의 판정 기준은 대시보드 코드와 배포 결과에 한정합니다.
- 실제 secret, token, 환경 변수 값은 조회하거나 노출하지 마세요. 필요한 경우 변수 존재 여부와 영향만 언급하세요.
- 루트 원인이 확인되지 않으면 추정으로 표시하고, 확인 방법을 별도로 제시하세요.

## 산출물 형식
- P0/P1/P2 순서로 findings를 먼저 제시.
- 각 finding에는 파일명, 가능한 경우 줄 번호, 영향, 재현 또는 확인 근거, 수정 제안을 포함.
- 확인된 사실과 추정은 분리.
- 대시보드 파일 외 Apps Script/시트 수정 제안은 "후속 제안"으로만 분리하고, 이번 작업 범위에는 포함하지 말 것.
